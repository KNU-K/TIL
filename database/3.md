## 기존 파일 시스템의 문제점

지금은 거의 사용하지않지만, 과거에는 응용 시스템을 개발하기 위해 파일 시스템을 사용하곤 했다.

- 데어터 종속성의 문제점

  Application Program과 파일시스템 간의 의존성이 높기 때문에 다른 Program에서 자신과 의존성이 없는 파일시스템에 접근할 수 없었다. 데이터의 구성이나 접근 방법을 변경하기 위해 해당 Application Program도 함께 변경되어야했다.

- 데이터 중복성의 문제점

  위에서 말한 종속성의 문제로 의존도가 올라가면서, 다른 Data File에 접근이 어려워 같은 데이터 일지라도 중복되는 데이터가 많이 생겼다.

## 데이터 중복성이 나쁜 이유

- consistency (일관성)

  데이터간의 불일치가 발생될 수 있다는 의미이다.

  > 김철수라는 학생이 2학년이라고 가정하자.

  > 이 때, 김철수의 모든 데이터의 학년이 2학년으로 되어있어야한다.

  > 하지만, 한개라도 3학년으로 잘못 표기 되어있게 되면, 데이터의 일관성이 무너진다.

- security (보안성)

  같은 데이터가 여러 군데 중복되어 저장되어 있다면 똑같은 보안을 장담할 수 없다.

  > 서랍에 있는 데이터와 PC에 있는 데이터의 보안을 같은 수준으로 유지하기란 상당히 어렵다.

- economics (경제성)

  데이터의 중복 저장으로 인해서 추가적인 공간이 필요하다.

  > 장치가 저렴해짐에 따라, 추가적인 저장 공간을 위한 비용이 큰 문제가 되지않는다.

  > 하지만, 만일 데이터에 대한 변경 발생 시, 중복 저장되어있는 모든 데이터를 수정해야한다.(사람이 직접 수정해야 하므로, 인건비에 의한 갱신 비용이 문제가 된다.)

- integrity (무결성 : 정확성)
  데이터가 사용자에 의도에 맞게 충분히 정확해야한다.

  > 이메일 데이터가 있다고 치면, example@email.com의 정규식의 형태를 띄어야한다.

  > 하지만 example이라고만 등록이 되거나, 특수문자가 허용되지않는 부분에서 특수문자를 넣게 되면 무결성이 위배가 되게 된다.

## DBMS (DataBase Management System)

기존의 화일 시스템에서 야기되는 데이터의 종속성과 중복성 문제를 해결하기 위해 제안된 시스템이라 할 수있다.

> 데이터베이스를 정의, 생성, 조작, 공유할 수 있도록 편리한 기능을 제공하는 SW System이다.

<img src="https://zitoc.com/wp-content/uploads/2019/09/Relationship-of-DBMS-with-Application-Program.png">

DBMS 가 DB와 Application Program 사이의 중간 역할을 해주게 된다. Application Program은 n개(여러개)가 될 수 있으며, 여러개의 프로그램이 하나의 Database를 DBMS를 통해 공유하게 된다. 이를 통해 종속성과 중복성을 최소화 할 수 있게 된다.

## DBMS의 필수 기능

- 정의(definition) 기능

데이터베이스 구조를 정의할 수 있는 기능 **DDL**을 사용한다.

> 데이터베이스와 테이블을 만들고 정의한다.

- 조작(manipulation) 기능

데이터베이스 접근하고 조작할 수 있는 기능 **DML**을 사용한다.

> 데이터베이스의 데이터의 검색, 삽입, 삭제등 연산 작업을 수행하도록 지원한다.

- 제어(control) 기능

데이터베이스의 정확성과 안전성을 유지할 수 있는 기능 **DCL**을 사용한다.

> 데이터 갱신을 정확하게 수행하거나, 허가된 사용자만 데이터에 접근할 수 있도록 보안을 유지하고, 정확하 결과가 유지되도록 병행 제어를 할 수 있도록 한다.

## DBMS의 장단점

- 장점

  1. 데이터 중복을 최소화할 수 있다.

     > 통제된 중복만을 허용하고, 불필요한 중복은 관계 설정으로 제거 할 수 있다.

  2. 데이터를 공유할 수 있다.

     > DBMS를 사용함으로써, 서로 다른 사용자가 각자의 목적에 맞게 공유가 가능하다.

  3. 데이터의 일관성을 유지 할 수 있다.

     > CASCADE 같은 옵션으로 다량의 데이터의 변경이 일어날 때, 발생되는 데이터 불일치를 최소화 할 수 이싿.

  4. 데이터의 무결성을 유지할 수 있다.

     > 데이터의 정확성을 유지할 수 있다. (타입에 맞게 사전 체크 가능)

  5. 데이터의 보안을 보장할 수 있다.

     > 정당한 사용자에게 허용된 데이터 연산만을 지원한다.

  6. 표준화를 달성할 수 있다.

     > 중앙 관리부서(root)에서 모든 요구 사항들을 전체적으로 수집, 분석해서 통합관리가 가능하다. 표준화로 효율적인 자료 관리가 가능하다.

  7. 데이터의 독립성을 제공할 수 있다.

     > 구현사의 세부 사항을 사용자에게 숨김으로써, 세부 사항보다는 일반적인 구조(데이터 구조)에만 집중하게 되어서 편리성과 효율성을 극대화 할 수 있다.(사용자 중심적)

- 단점

  1. 운영비가 증가할 수 있다.

     1. ORACLE과 같은 DBMS는 고가의 소프트웨어 제품으로 운영비가 증대될 수 있다.
     2. 유지 보수 비용도 만만치 않다.

  2. 자료 처리 방법이 복잡하다.

     1. 복잡한 구조를 이해하고 응용프로그램을 작성하기 위해서는 고급 프로그래머를 필요로 한다.

  3. 백업과 회복 기법이 어렵다.

     1. 장애 발생 시 backup과 recovery 기법이 복잡하다.
     2. a번으로 인한 신속한 복구가 어려울 수 있다.

  4. 시스템이 취약할 수 있다.
     1. 중앙관리에 의해 데이터의 효율성은 분명 있으나, 만일 장애 발생 시 일부의 장애라도 전체 시스템이 마비될 수도 있다.

## 데이터 독립성

데이터의 논리적인 구조나 물리적인 구조가 반영되더라도 응용 프로그램에는 영향을 주지 않는 것을 말한다.

- 논리적 데이터 독립성
  조직 전체의 논리적 구조를 변경 시키더라도 기존 응용 프로그램들에는 영향을 주지않는 능력

- 물리적 데이터 독립성
  저장 장치나 새로운 접근 기법의 개발로 성능 개선을 위해 물리적 구조를 변경하더라도, 이를 이용하는 응용 프로그램들에는 영향을 주지않는 능력이
